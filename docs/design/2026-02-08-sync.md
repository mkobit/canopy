## 1. Principles

Sync is event replication.
Every device should eventually have every event.
That is the entire requirement.

The event log is the source of truth.
`GraphEvent` objects keyed by `EventId` are what get replicated between devices.
Nodes and edges are a materialized view derived from the event log; they are also replicated as a convenience for fast local reads.

The current implementation uses [Yjs](https://yjs.dev/) as the CRDT substrate for replication.
The event log (`GraphEvent` objects) and the materialized graph (nodes and edges) are stored inside a `Y.Doc` as `Y.Map` shared types.
Yjs handles merge semantics and the binary sync protocol between devices.
This does not change the conceptual model: events remain the source of truth, and the `Y.Map` keyed by `EventId` preserves the set-reconciliation semantics of the original design.

---

## 2. What sync must do

The sync layer has four responsibilities:

1. **Replicate events**: transfer `GraphEvent` objects from one device to another so that every device eventually has every event.
2. **Deduplicate**: recognize that events with the same `EventId` are the same event, regardless of how many sources deliver them.
3. **Work offline**: events created while offline are stored locally and synced when connectivity is available.
4. **Be transport-agnostic**: different deployment scenarios use different transports. The sync layer adapts via the `SyncProvider` interface.

The sync layer does not:

- Order events for projection (ordering is by UUIDv7 timestamp, determined at event creation).
- Validate events (validation happens before an event enters the local log).
- Understand event semantics (it treats events as opaque payloads keyed by `EventId`).

## 3. Sync model

### The event log as a replicated set

The event log is logically a **set of events keyed by `EventId`**.
In the current implementation this set is a `Y.Map<StorableGraphEvent>` stored inside a `Y.Doc`.
The `Y.Map` key is the `EventId` (a UUIDv7 string); this gives automatic deduplication — setting the same key twice is a no-op.

When events are read back for projection they are sorted by `EventId`, which is lexicographically equivalent to creation time because `EventId` is a UUIDv7.

The materialized graph (nodes and edges) lives in two additional `Y.Map` instances on the same `Y.Doc`, updated whenever events are applied.
These maps exist for read performance; they are not the source of truth.

### Sync flow

Sync between two devices is Yjs state vector exchange:

1. Device A sends its state vector (a compact summary of which Yjs updates it has) to Device B.
2. Device B computes which updates A is missing and sends them.
3. Device A applies the updates to its local `Y.Doc`; Yjs merges them automatically.
4. The process runs in both directions simultaneously.
5. Presence state is exchanged separately via the Yjs Awareness protocol.

Because all three maps (events, nodes, edges) live in the same `Y.Doc`, a single Yjs sync pass replicates the full graph state.

### CRDT merge semantics

Yjs applies Last-Write-Wins semantics at the `Y.Map` key level when two devices concurrently set the same key.
For the events map this is safe: events are immutable after creation, so two devices setting the same `EventId` key will always write identical payloads.
For the nodes and edges maps, concurrent writes to the same node or edge key resolve via Yjs LWW, consistent with the LWW conflict resolution described in the [event system](2026-02-08-event-system.md).

## 4. Transport adapters

The sync engine is independent of the transport mechanism.
All transports implement the `SyncProvider` interface, which wraps the `Y.Doc` and `Awareness` instances and exposes `connect`, `disconnect`, and connection status events.

The exact wire format is transport-specific; the sync engine only requires that the transport can exchange Yjs binary updates and awareness updates.

### In-memory provider

An in-memory provider is available for testing.
It simulates a network by directly broadcasting Yjs updates between provider instances that share the same room name.

### File-based sync

Events can be replicated via a shared folder (e.g., Google Drive, Dropbox, or any filesystem sync tool).
Because events are stored in a `Y.Map` keyed by `EventId`, a file-based transport can serialize and exchange the Yjs binary state.
Alternatively, individual event files per device (with `EventId` as the filename) remain valid: a file-based adapter can read new files and apply them as Yjs updates.
This requires no hosted infrastructure and works for single-user multi-device scenarios.

### WebSocket / real-time

A WebSocket connection between devices or through a relay server.
Yjs updates are pushed in real-time as they are produced.
State vector exchange runs on connection establishment; incremental updates flow during the session.

### HTTP API

A pull-based model where devices poll a server for new Yjs updates.
The server stores the encoded `Y.Doc` state and serves incremental updates on request.
Devices push new updates to the server.

### LAN / peer-to-peer

Direct device-to-device sync over a local network.
No server required.
Useful for home setups or offline environments.

## 5. Offline-first

The system is offline-first by default.
The local `Y.Doc` is always available for reads and writes regardless of connectivity.

When offline:

- New events are added to the local events `Y.Map` and the materialized node/edge maps are updated.
- The user experience is unaffected.

When connectivity is restored:

- The sync provider exchanges state vectors with remote devices or servers.
- Missing Yjs updates are applied to the local `Y.Doc`.
- Yjs automatically merges the offline changes with the remote state.

The user may see their graph state change after sync if remote events merge with local ones.
This is expected behavior, not an error.

---

## 6. Conflict handling

The sync layer does not resolve conflicts at the event level.
Events are immutable; two devices writing the same `EventId` always produce identical payloads, so there is no conflict in the events map.

For the materialized nodes and edges maps, concurrent writes to the same key are resolved by Yjs LWW.
This is consistent with the timestamp-based LWW semantics in the [event system](2026-02-08-event-system.md).

The application can detect potential logical conflicts for UX purposes by comparing event timestamps: if a local event and a remote event both update the same property, the one with the later timestamp wins.
The full event log is available on every device, so the user can inspect history to see what was overridden.

## 7. Security and trust

### Event authenticity

Events received during sync must be validated before being used by the projection layer.
Validation includes structural checks, type conformance, and namespace rules (see [event system](2026-02-08-event-system.md), section 6).

> **Open question**: whether events should carry cryptographic signatures for authenticity.
> Signing adds complexity but is important for multi-user or untrusted transport scenarios.

### Transport encryption

Transports should use encryption in transit (TLS for WebSocket/HTTP, encrypted filesystem sync).
This is a transport concern, not a sync protocol concern.

### Document privacy

The `Y.Doc` contains the full graph state.
Transports must not expose binary updates to unauthorized parties.
End-to-end encryption (updates encrypted before leaving the device, decrypted only by authorized devices) is desirable for untrusted transports.

> **Open question**: encryption model for the `Y.Doc` state at rest and in transit.

## 9. Open questions

1. File-based sync format: individual event files per device vs. serialized `Y.Doc` state vs. hybrid.
2. Whether to replace Yjs with a bespoke event-log exchange protocol once the sync requirements are fully understood (Yjs was chosen to get multi-device sync working quickly; the `Y.Map<EventId>` model is compatible with a future custom set-reconciliation protocol).
3. Cryptographic event signing for authenticity in multi-user scenarios.
4. End-to-end encryption model for the `Y.Doc` at rest and in transit.
5. How to handle initial sync of a large vault to a new device (full `Y.Doc` snapshot vs. incremental update stream).
6. Whether to support selective sync (sync only certain namespaces or node types), which is difficult with a single monolithic `Y.Doc`.
