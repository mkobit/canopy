## 1. Principles

Sync is handled via CRDTs using Yjs.
Every device should eventually converge to the same state.
The sync layer leverages Yjs binary state vectors and updates to reconcile differences between clients.

The sync layer wraps a shared Y.Doc.
It does not manually resolve conflicts or write custom reconciliation logic.
It relies on Yjs to merge updates deterministically.
Conflict resolution and presence are handled natively by Yjs and its awareness protocol.
The sync layer acts as an engine and data store that transport adapters connect to.

---

## 2. What sync must do

The sync layer has four responsibilities:

1. **Replicate state**: transfer Yjs binary updates from one device to another so that every device eventually has the same document state.
2. **Provide presence**: use the Yjs awareness protocol to share online status and cursor positions.
3. **Work offline**: updates created while offline are stored locally and synced when connectivity is available.
4. **Be transport-agnostic**: different deployment scenarios use different transports through the SyncProvider interface.

The sync layer does not:

- Manually order events.
- Resolve conflicts with custom logic.
- Dictate the transport mechanism.

---

## 3. Sync model

### The shared document

The core of the sync model is a single Y.Doc.
This document contains the entire replicated state.
Sync is state vector reconciliation: each device computes which updates the other is missing and transfers them using standard Yjs sync protocols.

This model has useful properties:

- **Idempotent**: receiving the same update twice has no effect.
- **Order-independent**: updates can arrive in any order and will converge to the same state.
- **Automatic resolution**: conflicting updates are handled deterministically by the Yjs CRDT algorithm.

### Storage via Y.Map

Data is stored in the Y.Doc using Y.Map shared types.
The GraphStore manages three primary maps:

- `nodes`: stores all graph nodes.
- `edges`: stores all graph edges.
- `events`: stores the graph event log.

Nodes, edges, and events are serialized and stored directly in these maps.

### Sync flow

When two devices sync:

1. Device A advertises its state vector to Device B.
2. Device B computes which updates it has that A does not.
3. Device B sends the missing updates to A.
4. Device A applies the updates to its local Y.Doc.
5. The process runs in both directions.
6. Awareness state (presence) is also exchanged and updated.

## 4. Transport adapters

The sync engine is independent of the transport mechanism.
Different deployments use different transports via the SyncProvider interface.

### The SyncProvider interface

All transports implement a common SyncProvider interface.
A provider wraps the Y.Doc and Awareness instances.
It implements the following capabilities:

- **Connect**: establish a connection to the network or peer.
- **Disconnect**: tear down the connection.
- **Status events**: emit events when the connection status changes.

The exact wire format and protocol details are transport-specific.
The sync engine interacts with all transports through this interface.
For example, an in-memory provider is used for testing and simulates a network by directly exchanging Yjs updates between instances.

---

## 5. Offline-first

The system is offline-first by default.
The local Y.Doc is always available for reads and writes, regardless of connectivity.

When offline:

- New data is added to the local Y.Doc.
- The user experience is unaffected.

When connectivity is restored:

- The sync provider exchanges state vectors with remote devices or servers.
- Missing updates are applied to the local document.
- Yjs automatically merges the offline changes with the remote state.

The user may see their graph state change after sync if remote updates merge with local ones.
This is expected behavior, not an error.

---

## 6. Conflict handling during sync

The sync layer relies on Yjs for conflict resolution.
Because all state is stored in Y.Map structures, Yjs applies Last-Write-Wins semantics at the key level.
If two devices update the same key in a map while offline, the update with the higher client ID or later logical timestamp wins when they sync.

This means the application does not need custom logic to resolve diverging state.
The CRDT guarantees that all connected clients will eventually see the exact same document.

---

## 7. Security and trust

### Transport encryption

Transports should use encryption in transit.
This is a transport concern, not a sync protocol concern.

### Document privacy

The Y.Doc contains the entire graph state.
Transports must not expose binary updates to unauthorized parties.
End-to-end encryption (updates encrypted before leaving the device, decrypted only by authorized devices) is desirable for untrusted transports.

## 9. Future direction

A pure event-replication approach (with no CRDT dependency) was the original design intent.
The long-term architecture envisions sync as simple set reconciliation of events keyed by EventId, where projection handles conflict resolution.
However, Yjs was chosen for the initial implementation to get multi-device sync and presence working quickly and reliably.
In the future, the sync layer may be rewritten to replace the Yjs dependency with a bespoke event log exchange protocol.

---

## 10. Open questions

1. End-to-end encryption model for Yjs binary updates.
2. How to handle sync with very large documents (initial sync of a mature vault to a new device).
3. Whether to support selective sync (sync only certain nodes or namespaces), which is difficult with a single monolithic Y.Doc.
